# 2026-04-15 — Freshdesk 分诊系统修复

## 发现的问题

### 1. Rate Limit 导致工单未处理但 state 已标记（最严重）
- **现象**：apply_triage.js 把工单标记为"已关闭"写入 state，但 Freshdesk API 返回 429，实际未关闭
- **根因**：脚本没有检查 HTTP 状态码，不管成功失败都写 state
- **影响范围**：~100 张工单，跨多天累积
- **修复**：
  - request() 加了 429 指数退避重试（2s/4s/8s，最多3次）
  - 关闭/分派操作前检查 `_httpError`，失败不写 state
  - 请求间隔从 400ms 加到 800ms
  - 新增验证步骤：apply 完成后回查 Freshdesk 确认生效

### 2. Tag 名含中文/特殊字符导致 HTTP 400
- **现象**：agent 生成的 stage 字段含中文和长字符串，作为 Freshdesk tag 时报 400
- **修复**：cron prompt 里要求 stage 用英文简短格式（如 `order-shipping` 而非 `订单物流-包裹滞留德国`）

### 3. Cron 报告格式太简略
- **修复**：改为详细格式，按类别列每张工单号

### 4. Cron 报告投递
- **之前**：通过 send-feishu.py 推送到旧飞书频道 + announce 推送
- **现在**：只通过 announce 推送到当前群 chat:oc_dccd730144926488eb2afe86ae0d8a37

### 5. 飞书私聊配对码
- **修复**：dmPolicy 从 pairing 改为 open，allowFrom 设为 ["*"]

## apply_triage.js 修改要点
- request() 支持 429 重试 + HTTP 状态码标记
- 关闭操作前检查 closeRes._httpError
- 分派操作前检查 assignRes._httpError 和 tagRes._httpError  
- 新增验证步骤：处理完后回查 Freshdesk 确认 status/tags
- 验证失败的从 state 删除，下次重试
- 请求间隔 800ms

### 6. Tag 超过32字符导致 HTTP 400
- **根因**：Freshdesk 限制 tag 最长 32 字符，agent 生成的 reason 字段（如 `sender-spam: no-reply@mailer.shopify.com`，45字符）直接当 tag 用了
- **修复**：apply_triage.js 对 close 的 reason 和 assign 的 stage 都做 `.substring(0,32)` 截断
